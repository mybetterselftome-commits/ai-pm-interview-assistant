# AI 产品求职教练 V3 PRD

## 1. 产品定位

**产品名称：** AI 产品求职教练

**一句话定位：** 面向转行者的 AI 产品求职资产工作台，帮助用户完成能力盘点、JD 匹配分析、定向补强、模拟面试与复盘沉淀。

**目标用户：** 想转行 AI 产品岗 / Agent 产品经理 / AIGC 产品经理的人，尤其是已有运营、内容、项目、业务、教育、市场等非技术背景，希望快速建立 AI 产品求职竞争力的用户。

---

## 2. 核心洞察

转行 AI 产品的人，表面问题是“不懂 AI”“不会面试”，但真正的核心问题是：

> **无法把过往经历转化为目标 AI 产品岗位认可的能力证据。**

这类用户往往不是完全没有经验，而是存在以下困难：

1. 不知道自己适合 AI 产品经理、AI 产品岗、Agent 产品经理，还是 AIGC 产品经理。
2. 不知道如何把运营、内容、项目、业务经验迁移成 AI 产品岗位能力。
3. 看 JD 时知道自己“不够匹配”，但不知道具体差在哪里。
4. 学了很多 AI 概念，但不知道如何在面试中讲成产品方案。
5. 每次模拟面试或真实面试后，没有沉淀为可复用的求职资产。

因此，产品不应该只是一个“AI 问答助手”，而应该是一个帮助用户持续沉淀求职材料的工作台。

---

## 3. V2 到 V3 的迭代逻辑

### V2：AI 面试助手

V2 主要解决：

- AI PM 知识点多、散；
- 用户需要快速提问和学习；
- 系统能记录用户问过哪些知识点。

V2 已验证的能力：

- DeepSeek API 调用；
- Streamlit Web 部署；
- 设备 ID 与历史记录；
- 知识点体系与掌握度追踪。

### V2 的不足

1. 更像问答工具，而不是完整求职系统。
2. 个性化主要停留在“记录问过什么”。
3. 没有 JD 差距分析。
4. 没有面试评分标准。
5. 没有把输出沉淀成求职资产。

### V3：AI 产品求职资产工作台

V3 将产品主线升级为：

```text
能力盘点 → JD 匹配矩阵 → 模拟面试 Rubric 复盘 → 求职知识库 → 我的资产
```

核心变化：

- 从“问答”升级为“求职任务流”；
- 从“生成建议”升级为“沉淀资产”；
- 从“模型自由评价”升级为“Rubric 约束评价”；
- 从“泛分析 JD”升级为“岗位要求-用户证据匹配矩阵”；
- 从“单次输出”升级为“反馈闭环”。

## 4. 核心功能

## 4.1 能力盘点：生成个人 AI 产品求职画像

### 用户输入

- 过往背景；
- 目标岗位方向；
- AI 基础水平；
- 项目 / 产品 / 运营经历；
- 当前最大困惑。

### 系统输出

1. 一句话判断：用户当前最适合的 AI 产品方向；
2. 可迁移优势表：过往经历 → 可迁移能力 → AI 产品岗位价值；
3. 主要短板表：短板 → 面试风险 → 补强动作；
4. 求职准备度评分：
   - 产品方法；
   - AI 技术理解；
   - 数据指标；
   - 项目证据；
   - 面试表达；
5. 30 天补强优先级；
6. 面试包装建议。

### 产品价值

帮助用户从“我不知道自己能不能转 AI 产品”变成“我知道自己该从哪个方向切入”。

---

## 4.2 JD 匹配矩阵：把岗位要求转化为能力证据

### 用户输入

- 目标岗位 JD；
- 用户背景简述。

### 系统输出

核心输出是：**岗位要求-用户证据匹配矩阵**。

| JD 要求 | 面试官想验证什么 | 用户现有证据 | 匹配度 | 面试风险 | 补强建议 |
|---|---|---|---|---|---|
| 熟悉 RAG | 是否理解检索增强生成及其产品边界 | 有 AI 工具使用经验，但缺少 RAG 项目 | 中/低 | 容易被追问技术方案 | 准备一个 RAG 产品方案案例 |

此外还输出：

1. 岗位核心结论：冲刺 / 匹配 / 保底；
2. 三个最高风险短板；
3. 简历修改建议；
4. 高概率面试追问。

### 产品价值

不再泛泛告诉用户“你适合/不适合”，而是明确：岗位要什么、你有什么证据、缺什么、面试风险在哪里。

---

## 4.3 模拟面试：Rubric 评分复盘

### 用户输入

- 目标岗位方向；
- 系统抽取的面试题；
- 用户回答。

### Rubric 评分维度

系统固定按照 5 个维度评分，每个维度 20 分，总分 100：

| 维度 | 分值 | 评估重点 |
|---|---:|---|
| 结构表达 | 20 | 是否有背景、行动、结果、反思 |
| 产品思维 | 20 | 是否讲清用户痛点、产品方案、取舍 |
| AI 技术理解 | 20 | 是否理解模型、RAG、Agent、评估等概念 |
| 业务指标意识 | 20 | 是否有指标、数据、效果和 ROI 意识 |
| 复盘与取舍 | 20 | 是否能讲风险、失败、边界和下一步优化 |

### 系统输出

1. 总分；
2. 五维评分表；
3. 最大问题；
4. 面试官可能追问；
5. 优化版回答框架；
6. 可保存的 STAR 答案草稿。

### 产品价值

不让 AI “凭感觉说不错”，而是用评分标准约束模型，帮助用户知道自己到底弱在哪里。

---

## 4.4 求职知识库：面向面试场景的知识解释

### 当前 MVP 边界

当前版本不是 RAG 知识库，而是：

> **结构化知识目录 + 大模型面试场景解释。**

不会过度包装成“已实现 RAG”。

### 知识库分类

1. AI 技术理解；
2. AI 产品设计；
3. AI 运营增长；
4. 面试表达。

每个知识点输出：

1. 基础解释；
2. 为什么 AI PM 面试会问；
3. 产品案例；
4. 面试回答模板；
5. 可能的追问。

### 后续升级方向

V4 可升级为真实 RAG：

```text
AI PM 面试资料 / JD 样本 / 项目复盘文档
→ 文档切片
→ 向量检索
→ 生成回答并附引用来源
```

---

## 4.5 我的求职资产库

### 当前 MVP 能力

当前版本支持会话内保存以下资产：

- 求职画像；
- JD 匹配报告；
- 面试复盘；
- 知识库笔记；
- 补强计划。

并支持下载全部资产为 Markdown。

### 当前边界

当前资产库基于 `session_state`，属于 **MVP 会话级资产沉淀**。刷新页面或更换设备后不会长期保存。

### 后续升级

下一步需要接入数据库，结构化保存：

- 用户画像记录；
- JD 分析记录；
- 面试复盘记录；
- STAR 答案库；
- 反馈记录；
- 补强任务完成状态。

---

## 5. 反馈闭环

每次 AI 输出后，用户可以反馈：

- 有帮助；
- 太泛；
- 不准确；
- 保存为资产。

### 反馈价值

反馈用于识别：

1. 哪类输出真正有用；
2. 哪些 Prompt 容易产生空话；
3. 哪些回答存在 Bad Case；
4. 后续如何优化 Prompt 和评分规则。

当前版本先做会话内反馈记录，后续可沉淀到数据库，用于 Prompt 版本迭代和质量评估。

---

## 6. MVP 边界

### 已实现

1. 能力盘点；
2. JD 匹配矩阵；
3. 模拟面试 Rubric 评分；
4. 求职知识库；
5. 我的求职资产库；
6. 反馈闭环；
7. 示例数据填充；
8. Markdown 导出。

### 暂不做

1. 登录注册；
2. 长期数据库保存；
3. 简历文件上传解析；
4. 真正的 RAG 检索；
5. 多轮 Agent 面试；
6. 自动投递管理；
7. 支付和商业化。

### 为什么暂不做

因为当前阶段优先验证核心假设：

> 用户是否需要一个能把背景、JD、面试回答和知识补强串起来的 AI 求职资产工作台？

而不是一开始就堆复杂技术。

## 7. 产品指标设计

### 激活指标

- 首次能力盘点完成率；
- 3 分钟内生成求职画像比例；
- 示例数据使用率。

### 核心任务指标

- JD 匹配矩阵生成率；
- 模拟面试提交率；
- 面试复盘保存率；
- 求职资产保存数；
- Markdown 导出次数。

### AI 输出质量指标

- 有帮助反馈率；
- 太泛反馈率；
- 不准确反馈率；
- 输出可执行性评分；
- 面试复盘采纳率。

### 长期结果指标

- 简历修改完成率；
- 面试邀约转化率；
- 模拟面试分数提升；
- 7 日回访率；
- 用户最终求职结果。

---

## 8. 与 ChatGPT 的差异

普通 ChatGPT 是开放问答，用户需要自己知道该问什么。

本产品的差异是：

1. **任务结构化：** 将 AI 产品求职拆成能力盘点、JD 匹配、模拟面试、知识补强、资产沉淀。
2. **输出结构化：** 画像、矩阵、Rubric、STAR 答案、补强计划。
3. **资产沉淀：** 支持保存和导出求职材料。
4. **反馈闭环：** 收集“有帮助 / 太泛 / 不准确”，为后续 Prompt 优化提供依据。
5. **面向 AI PM 场景：** 不是通用求职助手，而是围绕 AI 技术理解、产品设计、运营增长和面试表达展开。

---

## 9. 面试展示话术

> 我最初做的是一个 AI PM 面试助手，但后来发现，转行者真正困难不是单个知识点不会，而是不知道如何把过往经历转化为目标岗位认可的能力证据。所以我把产品从 V2 的问答助手升级成 V3 的 AI 产品求职资产工作台。  
>  
> V3 的核心链路是能力盘点、JD 匹配矩阵、模拟面试 Rubric 复盘、求职知识库和我的资产。它的价值不是替用户生成一段泛泛建议，而是帮助用户把背景、岗位要求、面试回答和知识补强沉淀成可复用的求职材料。  
>  
> 当前 MVP 已实现会话内资产沉淀、Markdown 导出和反馈闭环。后续我会优先接数据库做长期资产保存，再升级多轮面试 Agent 和 RAG 知识库。

---

## 10. 后续迭代路线

### V3.1：资产持久化

- 接入数据库保存画像、JD 报告、面试复盘；
- 增加 STAR 答案库；
- 增加补强任务完成状态；
- 支持历史记录查看。

### V3.2：多轮面试 Agent

- 面试官连续追问；
- 根据用户回答动态生成下一题；
- 根据 Rubric 汇总表现；
- 输出长期能力变化。

### V3.3：Prompt 版本化与评估

- 抽离 Prompt 模板；
- 增加 Prompt 版本号；
- 建立典型 Case 测试集；
- 对比不同 Prompt 输出质量。

### V4：RAG 知识库

- 建立 AI PM 面试知识文档；
- 支持向量检索；
- 回答附引用来源；
- 根据用户短板推荐知识点。
